# コントリビューションガイド（日本語）

本Organization配下のリポジトリにご関心をお寄せいただき、ありがとうございます。

本ドキュメントでは、Organization 全体に適用される**開発ルール**について説明します。
各リポジトリで特別な記載がない限り、本ガイドに従ってください。

> [!IMPORTANT]
> コミュニティ活動の立ち上げ期間中のため、下記内容には予告無しの変更があることをご了承下さい。

## ブランチ戦略

本Organizationでは、Git Flowをベースに以下の命名規則を標準として運用します。

### ブランチ名

- main
  - 常にリリース可能な状態を維持します
  - 直接のコミットは禁止されています
- develop
  - 日常的な開発の統合先ブランチです
- feature/\*
  - 新機能追加や改善作業用のブランチ
  - developから作成してください
- hotfix/\*
  - リリース済みバージョンへの緊急修正用
  - mainから作成してください

featureやhotfixは、変更対象のカテゴリごとに階層を作っても問題ありません。
例: feature/meridian-lite/XXX-module

> [!NOTE]
> コミュニティで管理しているレポジトリの開発・変更をリクエストする場合は、まず個人のレポジトリにフォークしてください。
> そしてfeatureやhotfixなどのブランチを作ったうえで、これをプルリクエストしてください。

### タグ名

- vX.Y.Z
  - 動作確認がとれたものとして固定バージョンです
  - mainから作成してください

## 変更の取り込みとマージ

すべての変更はPull Request (PR)経由で行ってください。
PRのマージ先ブランチは以下のルールを守ってください

- feature/* &rarr; develop
- hotfix/* &rarr; main

またPRはテンプレートに従って、以下の情報を必ず記載してください

- 変更内容の要約
- 動作確認・テスト方法
- 関連 Issue（存在する場合）

原則としてPRは、通常のマージとして取り込みます。
各リポジトリ設定に応じて、レビュー承認を実施し、レビューの人数やアサイン、スコープはコミュニティ内で都度相談して決定します。

## コーディングルール

原則としてレポジトリには、formatterやlinterツールを加える方針です。
C++でのプログラムは[.clang-format](https://github.com/Ninagawa123/Meridian_LITE/blob/main/Meridian_LITE_for_ESP32/.clang-format)を基本的に採用します。
